# 人力甘特图 MVP 改造说明

## 产品经理视角

### 目标用户

- 中小团队的 PM、PMO、交付负责人
- 需要在一屏内看到资源占用、项目状态和近期风险的人

### 当前阶段的 MVP 定义

- 看得见：摘要卡、甘特时间线、项目清单、风险与里程碑
- 改得动：可编辑项目标题、负责人、状态、优先级、说明、里程碑、开始时间、持续时间、进度
- 存得住：修改结果自动保存到浏览器本地 `localStorage`
- 跑得起来：支持 `pnpm build` 和本地 Docker 部署

### 本轮优先级

1. 先让核心排期闭环可用，而不是继续停留在静态设计稿
2. 保留 Figma 风格视觉语言，但让所有操作都具备真实反馈
3. 首版先使用本地状态和本地存储，后续再接 API / 数据库

## UI / UX 视角

### 页面结构

- 左侧：产品定位、MVP 范围、交互原则
- 顶部：搜索、团队筛选、新建项目、导出
- 中部左侧：甘特图主视图 + 项目清单
- 中部右侧：详情编辑、风险与里程碑、活动记录

### 关键交互

- 搜索项目、团队、负责人
- 按团队和状态筛选
- 切换 7 / 10 / 14 天视窗
- 点击甘特条或项目清单进入详情编辑
- 新建项目并直接选中
- 所有修改显式保存，并写入活动记录

### 体验原则

- 一屏决策，不跳页
- 中文文案优先，降低理解门槛
- 先保证浏览、编辑和反馈，复杂拖拽留给后续版本
- 移动端采用信息堆叠，不强行保留桌面三栏

## 开发视角

### 当前实现

- React + TypeScript + Vite 单页应用
- 单文件状态管理，便于快速验证 MVP
- `localStorage` 持久化工作区数据
- Docker 通过 `node:22-alpine` 构建静态资源，再用 `nginx:1.27-alpine` 提供服务

### 后续建议

- 抽离 `workspace` 数据层
- 增加 REST API 或 GraphQL 接口
- 接入用户系统、权限、评论、附件与服务端导出
- 再往下一步再做多人协作和拖拽排期

## 测试策略

- 构建验证：`pnpm build`
- 代码检查：`pnpm lint`
- 容器验证：`docker build` + `docker run`
- 访问验证：本地 `http://127.0.0.1:8080` 返回 `200 OK`
